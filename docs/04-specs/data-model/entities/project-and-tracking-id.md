# Project and tracking ID

A toolkit "project" is a directory on disk with a recognisable shape. The engine identifies a project by:

- Presence of `m365agents.yml` (v3 + v4)
- Or legacy `.fx/projectSettings.json` (v3 historical)

## Tracking ID

Every project carries a stable identifier used for telemetry attribution and idempotent provision.

| Source | Field |
|--------|-------|
| Manifest | `id` (Teams App ID) — generated during scaffold |
| Env file | `TEAMS_APP_ID`, `TEAMS_APP_TENANT_ID` after first provision |
| Telemetry | `projectId` derived from manifest `id` (anonymised hash for telemetry) |

The tracking ID is **not** PII — it is the Teams App ID, which is meant to be world-public once published. Telemetry hashes it so we can count distinct projects without resolving them back to apps.

## Project name vs safe name

| Concept | Where |
|---------|-------|
| `appName` | Display name used in UI and manifest |
| `safeProjectName` / `SafeProjectName` | URL-safe, lowercase, dash-stripped projection used for Azure resource names |
| `SafeProjectNameLowerCase` | Lowercased variant for resources that forbid uppercase |

Generated by `getTemplateReplaceMap()` in [`packages/core-next/src/templates/scaffold/replaceMap.ts`](../../../packages/core-next/src/templates/scaffold/replaceMap.ts).

## Multi-environment

A project supports multiple environments. Each environment has its own env files (`env/.env.{name}` and `env/.env.{name}.user`). Common environments: `local`, `dev`, `prod`. Custom environments are added via `atk env add <name>`.

The lifecycle YAML is **shared across environments**; only the env files differ.
